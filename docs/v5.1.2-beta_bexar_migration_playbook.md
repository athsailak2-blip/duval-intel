## Why this playbook exists

The May 2026 universality audit on the v5.1.1-beta-seeded Bexar Phase 1-4 build identified 11 specific Bexar-specific leaks in `scaffold/pipeline/` and 4 in `dashboard/`. The framework code knew it was running for a specific county. v5.1.2-beta locks the Universality Contract (MASTER_PROMPT §4.31) and ships the architectural scaffolding (schema, translator registry, sale_date_rules, owner_name_patterns, regression test) needed to migrate the contaminated pipeline into a clean state.

This playbook is the operator-authorized path to clean the Bexar working tree IN-PLACE without throwing away the 287-lead production data, the BCAD enrichment, the dashboard work, or the operator-decision artifacts captured during Phase 0-4.

The current Bexar output becomes the **regression baseline**. If the cleaned framework cannot reproduce equivalent output, that is a v5.1.2-beta bug to fix BEFORE Phase 5 starts.

The scraper is responsible for normalizing source fields to framework-canonical lowercase names BEFORE writing JSONL. `translator_config` field-name references below use the NORMALIZED names the scraper writes to `raw_payload`, NOT the source's raw protocol attribute names.

The translator reads these lowercase normalized fields from `raw_payload`: `address`, `doc_number`, `recording_year`, `recording_month`, `city`, `zip`, `layer_id`. Bexar's foreclosure scraper already produces these canonical names, so no `field_map` is needed for the foreclosure source. (v5.1.2-beta-r3+: if any other county's foreclosure scraper produces different field names, add a `field_map` block on the source config — the translator resolves canonical names through `field_map` automatically.)

For the parcel-master source:

```json
"parcel_master": {
  ... existing fields ...
  "translator": "parcel_master",
  "parcel_id_prefix": "BCAD-",
  "field_map": {
    "address": "situs_address",
    "city": "situs_city",
    "zip": "situs_zip",
    "owner_mailing_address": "owner_mailing_addr1",
    "property_use": "property_class"
  }
}
```

The `field_map` block (v5.1.2-beta-r3+) tells the parcel_master translator how to find canonical fields when the scraper writes source-specific field names. Keys are canonical names the translator expects; values are the actual field names in `raw_payload`. Canonical fields not listed in `field_map` are read directly (identity mapping).

For Bexar specifically: 5 mappings are required (`address`, `city`, `zip`, `owner_mailing_address`, `property_use`). All other canonical fields the parcel_master translator reads (`parcel_id`, `owner_name`, `assessed_value`, `land_value`, `improvement_value`, `year_built`, `acres`, `legal_description`, exemption booleans) match Bexar's scraper output identically and need no mapping.

The parcel_master translator reads framework-canonical fields from `raw_payload` (resolved through `field_map` if present): `parcel_id`, `address`, `owner_name`, `owner_mailing_address`, `owner_mailing_city`, `owner_mailing_state`, `owner_mailing_zip`, `city`, `zip`, `assessed_value`, `land_value`, `improvement_value`, `year_built`, `property_use`, `acres`, `legal_description`, and pre-parsed exemption booleans `exempt_homestead`, `exempt_over_65`, `exempt_disabled`, `exempt_veteran`. If the scraper emits a raw `exemptions` string instead of (or in addition to) the booleans, supply `translator_config.exemption_codes` and the translator will hybrid-resolve: explicit booleans from raw_payload take precedence; missing flags are filled from the exemptions string.

**One-time data shape transform (v5.1.2-beta-r2 only):** Pre-r2 Bexar `data/raw/parcel_master.jsonl` records are FLAT (no `raw_payload` wrapper). Before running Step 5, transform the file into wrapped shape using this deterministic script:

```python
import json
from pathlib import Path

src = Path("data/raw/parcel_master.jsonl")
dst = src.with_name("parcel_master.wrapped.jsonl")
with src.open() as fin, dst.open("w") as fout:
    for line in fin:
        if not line.strip():
            continue
        record = json.loads(line)
        # Move parcel_id-derived identity to top-level metadata.
        raw_record_id = f"raw_{record.get('parcel_id', '').strip()}"
        wrapped = {
            "raw_record_id": raw_record_id,
            "source_id": "parcel_master",
            "source_url": record.get("source_url", f"about:blank/parcel_master/{record.get('parcel_id')}"),
            "source_fetched_at": record.get("scraped_at") or record.get("source_fetched_at"),
            "parser_confidence": record.get("parser_confidence", 100),
            "raw_payload": record,
        }
        fout.write(json.dumps(wrapped, separators=(",", ":")) + "\n")
src.unlink()
dst.rename(src)
```

This transform is DETERMINISTIC. It does not change any record's content; it only wraps each flat record into the canonical shape. Document in `runs/bexar_tx/operator_notes.md` as part of the migration. The 287-lead baseline is preserved because the underlying data is bit-identical inside the wrapper.

For `data/raw/foreclosure_notices_map.jsonl`, no transform is needed — Bexar's foreclosure scraper already wraps its output in `raw_payload` and uses lowercase normalized field names that match the contract.

**Note on actual scraper field names:** Read `scrapers/foreclosure_notices_map.py` and `scrapers/parcel_master.py` to confirm the on-disk normalized field names match the canonical names the translator expects. If a scraper uses non-canonical names (e.g. `situs_address` instead of `address`), either rename in the scraper output or — short-term — extend the translator config to accept a per-source `field_map` (added in v5.1.2-beta-final).

### Step 5 — Refactor build_leads.py to use config-driven translator dispatch

The current Bexar `scaffold/pipeline/build_leads.py` has hardcoded:

```python
fnm_path = raw_dir / "foreclosure_notices_map.jsonl"
parcel_master_path = raw_dir / "parcel_master.jsonl"
# ... and direct calls to translate_foreclosure_notices_map(...)
```

v5.1.2-beta wants this to be a config-driven loop:

```python
from scaffold.pipeline.translators import lookup as lookup_translator

county_config = json.load(open(f"config/counties/{county_slug}.json"))

all_signals = []
all_parcels = []
all_per_signal_meta = {}

for source_id, source_config in county_config.get("sources", {}).items():
    if not source_config.get("enabled", True):
        continue
    translator_name = source_config.get("translator")
    if not translator_name:
        # No translator wired — skip (raw data may still be pulled by adapter for inspection).
        continue
    source_config_with_id = dict(source_config)
    source_config_with_id["_source_id"] = source_id
    raw_path = raw_dir / f"{source_id}.jsonl"
    if not raw_path.exists():
        # Source not yet pulled. Continue with what we have.
        continue
    raw_records = [json.loads(line) for line in raw_path.read_text().splitlines() if line.strip()]
    translate_fn = lookup_translator(translator_name)
    signals, parcels, per_signal_meta = translate_fn(
        raw_records, county_config, source_config_with_id
    )
    all_signals.extend(signals)
    all_parcels.extend(parcels)
    all_per_signal_meta.update(per_signal_meta)
```

**Scrub** the hardcoded `build_label_reason` string (`"Source-limited build: only the Bexar County foreclosure_notices_map ..."`). Replace with config-driven prose composed from the actual source list:

```python
sources_used = [s for s, c in county_config["sources"].items() if c.get("enabled", True) and c.get("translator")]
build_label_reason = (
```

## What this playbook does NOT cover

- **Phase 5 scoping** — handled separately. Operator-volunteered guidance: clerk_recordings should exclude foreclosure-stage instruments (NSTS, NTS, NOD, Tax Foreclosure Notice — already covered by foreclosure_notices_map). Focus: lis pendens, federal/state tax liens, mechanics liens, judgment liens, affidavits of heirship, distribution deeds, executor/administrator deeds, quitclaim deeds, correction deeds, partition deeds. POAs at MEDIUM confidence; HOA liens at LOW confidence (~50); skip releases of lien and subordinations. PublicSearch's `foreclosures` category label is operator-known but Claude Code should still verify via recon (universality test of the framework's doc-type recon process).
- **publicsearch_clerk_recordings translator implementation** — when Phase 5 starts, a new built-in translator will be added to canonical `scaffold/pipeline/translators/` following the pattern established by the three existing built-ins.
- **Second-county test** — separate operator session. Once v5.1.2-beta-final ships, pick a Maricopa/Cuyahoga/Pinellas etc. county, run Phase 0 with a fresh county slug, verify zero framework code changes were needed.

## Failure-mode escape hatches

**If migration fails partway through:** the Bexar repo is git-versioned. Revert all uncommitted changes (`git checkout -- .`) and start over from Step 1.

**If output equivalence fails after Step 12:** treat it as a v5.1.2-beta canonical bug. Open an issue in the framework repo, fix the bug, re-pull canonical files, redo Steps 1-12. Do NOT ship contaminated framework code disguised as v5.1.2-beta because the migration was hard.

**If a step instruction is ambiguous:** stop, capture the ambiguity in `runs/bexar_tx/backlog/v5.1.2-beta-migration-ambiguities.md`, and ask the operator. The migration is a one-time event; getting it right matters more than speed.
