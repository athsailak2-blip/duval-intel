# v5.3.0 Patch Plan — Source of Record Engine + Build Semantics + Semantic Verification

## A. Version target and canonical base

- **Version target:** v5.2.0 → v5.3.0.
- **Canonical base:** the Bexar `main` branch is the de facto canonical harness for this
  patch cycle. Any uploaded zip is NOT the base — all patches target Bexar `main` and its
  working branch `sync/v5.3.0-source-of-record`.
- **Driver:** v5.3.0 hardens the recon → build pipeline against eight defect classes
  surfaced empirically during an El Paso County build run. Each gap is anchored to a
  concrete defect or finding (§B).

This patch plan is the only document in the v5.3.0 cycle permitted to reference
county-specific evidence (El Paso, EPCAD, ACT, specific lead counts). Every universal
framework file must stay county-agnostic; the county-agnostic regression scanner
(`scaffold/tests/test_county_agnostic_regression.py`) enforces this and is run as a gate.

## B. Empirical anchors — each gap mapped to its El Paso defect or finding

| Gap | Title | Empirical anchor (El Paso run) |
|---|---|---|
| Gap 1 | PDF / sample-document inspection | `foreclosure_notices` was classified DEFERRED at Phase 0 because the listing/index page exposed only instrument number + sale date + legal description. The actual notice PDFs carried street address, debtor name, deed-of-trust document number, sale date, and lender — all directly extractable. The deferral was a false negative; sample-document inspection would have caught it. |
| Gap 2 | Documented API discovery | EPCAD parcel data has a fully documented Postman API. Initial recon missed it — recon stopped at the HTML/WAF classification. The operator surfaced the API manually. A required API-discovery sub-step would have found it. |
| Gap 3 | Bulk-data availability classification | The ACT tax-collector portal allows per-account search only — no full delinquent-roll enumeration. Discovered during build, not recon. EPCAD's `streetName` search is also page-capped and suffix-sensitive — another per-record limitation discovered late. Both are recon-surfaceable constraints. |
| Gap 4 | Semantic verification | A "UMC" entity collapse, an aggregator double-merge, and 19 of 19 v5 audit cases showed that string-equal joins without semantic verification produce wrong merges. |
| Gap 5 | Debtor / party-role rules | Hospital lien, code lien, and federal/state tax lien records attributed leads to the wrong party — the filer (e.g. City of El Paso, the state workforce commission, a hospital, a mortgage servicer) was treated as the lead owner instead of the debtor. Filer-suppression rules were missing. |
| Gap 6 | Signal aggregation | A single parcel stacked 30 Hospital Lien chips; a "UMC" entity collapsed 148 signals into one row. Aggregation lacked dedup and per-type cap discipline. |
| Gap 7 | Build Mode Protocol | The build phase was implicit — no explicit gate between recon and build. An aggregator double-merge inflated the lead count to 1,962 before the defect was caught. |
| Gap 8 | Aggregator idempotency | v5 fix: the aggregator must read from a stable `clerk_leads_base.json` input, never from its own prior output — re-running it must be idempotent. |
| EPCAD-as-optional amendment | `foreclosure_notices` v4 | The `foreclosure_notices` source achieved 97% address coverage WITHOUT EPCAD enrichment. `resolution_status` must be decoupled from `enrichment_status` — a lead can be resolved (address known) without being enriched (parcel/valuation attached). |

## C. Session structure (locked)

The v5.3.0 cycle is split into five physical sessions. Session boundaries are hard units
— each session ends at a clean commit boundary with passing gates.

| Session | Scope | Gaps |
|---|---|---|
| A1 | Source of Record Recon foundation + wording realignment + schema | Gaps 1, 2, 3 |
| A2 | Lead Origination + Build Semantics + EPCAD-optional amendment | Gaps 5, 6, 8 |
| A3 | Semantic Verification Contract | Gap 4 |
| A4 | Build Mode Protocol — absorbs Patch 2 stash concepts WITHOUT applying the stash | Gap 7 |
| A5 | Final regression + tag v5.3.0 + cut release zip + stub honesty pass | — |

Note on session ordering: the operator reordered the cycle so Source-of-Record Recon
(A1) lands first — recon correctness is the foundation everything downstream depends on.

## E. Per-gap invariant requirements

Every gap ships with at least one invariant test under `scaffold/tests/v5_3_0/`,
committed only after it passes alongside the four existing gate suites:

- Gap 1 — `test_recon_requires_pdf_inspection.py`
- Gap 2 — `test_recon_requires_api_discovery.py`
- Gap 3 — `test_recon_requires_bulk_classification.py`
- Schema — `test_schema_source_of_record_matrix.py`
- Gap 5 — debtor/party-role invariant (A2)
- Gap 6 — signal-aggregation invariant (A2)
- Gap 8 — aggregator-idempotency invariant (A2)
- Gap 4 — semantic-verification invariant (A3)
- Gap 7 — build-mode-protocol invariant (A4)

A per-gap invariant must pass before its gap's commit lands. The four existing gate
suites (Golden path, County-agnostic regression, Atomic writer, Translator registry)
must continue passing after every commit.

## F. Stub honesty pass (Session A5)

`scaffold/ops/verify_live.py` and `scaffold/ops/watchdog.py` currently ship as stubs
(CLI surface only) per `MASTER_PROMPT.md §4.27`. Session A5 performs a stub honesty pass:
the file headers, `MASTER_PROMPT.md`, and `README.md` must accurately disclose that
these are stubs. **v5.3.0 does NOT implement real production verification or watchdog
runtime** — that is v5.4.0+ work. Documentation must not overclaim a capability the
framework does not have.

## G. Out-of-scope deferrals

- Bulk parcel index → deferred to v5.4.0.
- Daily-refresh template → deferred to v5.4.0.
- Real `verify_live.py` / `watchdog.py` implementation → deferred to v5.4.0+.

## H. Constraints

- The universality contract is enforced by the county-agnostic regression scanner — no
  county-specific token may appear in a universal file. This patch plan (`docs/`) is the
  sole exception and is scanner-exempt.
- The v5.2.0 Patch 2 stash stays parked — never popped, deleted, or blindly applied.
- Per-gap test pass is required before each gap's commit.
- Session boundaries are physical units — a session ends at a clean, gate-passing commit
  boundary.
- Every commit uses explicit `git add <path>` of named files — never `git add .` or
  `git add -A`.
