# v5.4.0 Session 6 — Part 1: Seam Design (matched_leads.json → scoring / dashboard)

**Status:** DESIGN — awaiting operator approval. Nothing in this document is
implemented. Part 2 executes the cutover only after the seam option below is
approved.

**Branch:** `feat/v5.4.0-pipeline-engine` (HEAD `877d313`).

## 0. Context

The v5.4.0 staged engine — §17 debtor party → §18 aggregation key + leads-base
writer → §19 idempotent aggregator → §20 semantic verification — is built and
proven end-to-end (Session 5). It emits `matched_leads.json` (per
`matched_lead_record.schema.json`) plus `evidence_ledger.json` and a §20 deploy
verdict.

Per operator decision (a), the monolith's **scoring, deal-path classification,
title-complexity, and dashboard-generation stages are retained** — they were
never the broken part — and must run **downstream of `matched_leads.json`**.
Session 6 connects the two. This document (Part 1) designs the connection; Part
2 executes it after approval.

The genuine problem this seam solves: the staged engine replaced the monolith's
*signal → identity → aggregation* core, but the scoring/classification/dashboard
stages still consume the monolith's pre-cutover `leads.json` shape. The two
shapes do not match.

## 1. Shape gap — field-level

### 1.1 What the scoring stage consumes today

`score.compute_score(stack, attributes)`, `classify.classify_deal_paths(stack,
attributes)`, and `build_leads._title_complexity(stack)` all consume a **`stack`
object** (built by `stack.stack_signals`) plus an **`attributes`** list:

| `stack` field | Type | Consumed by |
|---|---|---|
| `parcel_id` | str | all |
| `active_signals` | list of raw normalized-signal dicts — each with `normalized_doc_type` (UPPERCASE), `pattern`, `event_date`, `amount`, `source_url`, `doc_type_confidence`, `document_priority` | score (base), classify (doc-type tests), title-complexity |
| `pattern_set` | list[str] — distinct `pattern` values (`lien`, `foreclosure`, `estate`, `tax`, `code`, …) | score (stack bonus), classify |
| `stack_depth` | int | score, classify |
| `recent_flag` | bool — any active signal ≤30d | score (recency bonus) |
| `patterns` / `stack_contrib_patterns` | list[str] | lead assembly, dashboard chips |
| `amounts` | list[float] | collected; **not** read by `compute_score` |
| `attributes` (separate arg) | list[str] — `high_equity`, `vacant`, `absentee`, … | score (attribute bonus), classify |

The dashboard (`dashboard.project_lead`) additionally consumes a **parcel**
object: `situs_address`, `situs_city`, `situs_state`, `owner_name`,
`assessed_value`, `last_sale_price`, `last_sale_date`, `year_built`.

### 1.2 What `matched_leads.json` provides

`matched_lead_record` (`additionalProperties: false`): `lead_id`,
`primary_parcel_id`, `owner_name`, `owner_type`, `filer_entity`,
`review_reason`, `parcel_resolution_status`, `enrichment_status`, `signals[]`,
`source_ids`, `evidence_ids`, `lead_status`. Each `signals[]` item is a §18
**aggregated SignalGroup**: `aggregation_key`, `signal_type`,
`canonical_doc_type` (lowercase), `count`, `instrument_numbers`, `source_urls`,
`evidence_ids`, `source_ids`, `earliest_recorded_date`, `latest_recorded_date`,
`recorded_date_range`.

### 1.3 The gap

| # | Gap | Detail |
|---|---|---|
| G1 | **Doc-type namespace mismatch** | matched_lead `canonical_doc_type` is the lowercase per-county §17 taxonomy (`hospital_lien`, `executor_deed`, `foreclosure_notice`). `score.BASE_SCORE` and `normalize.CANONICAL` use the UPPERCASE framework taxonomy (`HOSPITAL_LIEN`, `EXECUTORS_DEED`, `NOTICE_OF_SALE`). The two are different sets at different granularities (one §17 `foreclosure_notice` vs. several monolith `NOTICE_OF_*`). The seam needs an explicit `canonical_doc_type → normalized_doc_type` bridge. |
| G2 | **No `pattern`** | scoring, classification, and stacking key off `pattern` (`lien`/`foreclosure`/`estate`/`tax`/`code`/…). The matched_lead carries no `pattern`. It must be derived from `canonical_doc_type` (the retained `normalize.CANONICAL` table carries `lead_pattern`). |
| G3 | **Aggregated groups vs. raw signals** | `stack.active_signals` is a list of *raw* per-signal dicts with per-signal lifecycle/TTL. matched_lead `signals[]` are §18 *aggregated* groups with a `count` badge. §18/§19 have already done dedup/collapse; the seam treats each SignalGroup as one stack entry (its `count` is the contribution). Per-signal `event_date` granularity is replaced by the group's `earliest/latest_recorded_date`. |
| G4 | **`attributes` absent** | the attribute bonus (score) and several classify branches need `high_equity`, `vacant`, `absentee`, `long_term_owned`, etc. These are **enrichment** data (parcel-master / GIS). The staged pipeline (§17-§20) does no enrichment — matched_lead carries only the `enrichment_status` flag, not the data. The seam needs an enrichment source for attributes. |
| G5 | **Parcel display data absent** | the dashboard needs `situs_address`, `owner_name`, `assessed_value`, `last_sale_price`, `year_built`. matched_lead has `primary_parcel_id` and `owner_name` only. The rest is parcel-master enrichment. |
| G6 | **`amounts` dropped** | `raw_event_record` carries `amounts`; the §17 engine does not carry them onto `debtor_resolved_record`, so they are absent from leads-base and matched_lead. `score.compute_score` does **not** read amounts (verified), so scoring is unaffected; the dashboard amount display would lose them. Low severity — flagged. |
| G7 | **F-8 / §09 signal shape** | the monolith lead's `signals` is a list of `signal_id` strings (§09 §4 shape). matched_lead `signals[]` is the §18 rich aggregated-group shape. Per Session 1 finding F-8 the new architecture uses the §18 shape; §09's `signals`-field description needs a reconciliation note. |

G1, G2, G3 are the core seam work. G4/G5 mean **an enrichment stage must sit in
the seam or just downstream of it** — this is the largest open question (see
§7).

## 2. Seam options

### Option X — scoring writes into the matched_lead record

Relax `matched_lead_record.schema.json` `additionalProperties: false`; add
`score`, `score_reasons`, `deal_paths`, `title_complexity_*`, `lead_status_history`
as optional fields the scoring stage populates.

- **Schema impact:** the §19 aggregator validates its output against this
  schema; the scoring fields must be optional or §19's own output fails
  validation. The matched_lead becomes a record two stages write.
- **§20 ordering:** §20 verifies `matched_leads.json`. If scoring then mutates
  the same record, the §20 verdict was computed against a now-changed record —
  "matched_lead" becomes ambiguous (pre- vs post-scoring).
- **Idempotency:** §19.D guarantees `matched_leads.json` is byte-identical on
  re-run. If scoring writes into the same file/record, `matched_leads.json` is
  no longer purely the §19 output — the clean §19 idempotency artifact boundary
  is lost.
- **Stage-owns-its-fields:** **violated.** The entire v5.4.0 architecture — and
  the F-1 ratification specifically — is "each pipeline stage owns its own
  fields." Option X has the scoring stage write the §19 stage's record.

### Option Y — scoring emits a new `scored_lead` record (recommended)

The scoring stage emits a **new downstream record type**, `scored_lead_record`,
that references the matched_lead by `lead_id` and carries `score`, `tier`,
`score_reasons`, `deal_paths`, `title_complexity_*`, `attributes`,
`review_flags`, `lead_status` / history. `matched_lead_record` is unchanged and
remains the immutable §19/§20 output of record.

- **Schema impact:** one new `scored_lead_record.schema.json`; existing schemas
  untouched. Mirrors the Session 1 pattern — `raw_event` → `debtor_resolved` →
  `leads_base` → `matched_lead`, each a distinct record owned by one stage.
  `scored_lead` is the natural next stage record.
- **§20 ordering:** §20 runs on `matched_leads.json` (the pure §19 output)
  **before** scoring. Clean: §20 gates identity/aggregation correctness;
  scoring runs after, only on verified matched leads. Matches §20.G and the
  pipeline order §17→§18→§19→§20→(seam)→scoring.
- **Idempotency:** `matched_leads.json` stays the pure §19 output (§19.D
  intact). `scored_leads.json` is the scoring stage's own deterministic
  artifact. Each artifact has exactly one owner.
- **Stage-owns-its-fields:** **honored** — the principle the whole v5.4.0
  architecture follows.

### Option Z — fold scoring into the dashboard-build stage

No separate scored record; the dashboard-build stage consumes `matched_leads.json`,
scores inline, and emits dashboard `data.json` directly.

- Fewer record types, but scoring is no longer independently inspectable or
  testable, and the §09 scored-lead schema collapses into the dashboard
  payload. Weaker for audit and for the golden-path's per-layer coverage.

### Recommendation — **Option Y**

Option Y is the only option that honors the staged-stage-owns-its-fields
principle that the v5.4.0 architecture (and the ratified F-1 finding) is built
on. It keeps `matched_leads.json` immutable as the §19/§20 output of record,
keeps §19.D idempotency's artifact boundary clean, and places §20 cleanly
before scoring. Option X makes the matched_lead a record two stages mutate,
muddles the §20 gate, and breaks the §19 idempotency artifact boundary. Option
Z sacrifices testability and audit granularity.

**Recommended pipeline-record chain:**
`raw_event` → `debtor_resolved` → `leads_base` → `matched_lead` → **`scored_lead`** → dashboard payload.

| `classify.py` | **RETAINED (downstream)** | deal-path classification — runs after the seam. |

| `evidence.py` | **REVIEW — likely RETIRED** | the monolith evidence helper; superseded by Session 5's `evidence_ledger.py`. Part 2 confirms no retained stage still imports it before retiring. |

Its `run_pipeline` signal→identity→aggregation core (`normalize_signal`, `derive_synthetic_signals`, `_apply_parcel_master_matching`, `_signals_by_parcel`, the signal→stack→lead orchestration) is **RETIRED** — replaced by §17-§19. monolith" must not catch them. The only genuinely RETIRED code is the
signal→identity→aggregation orchestration *inside* `build_leads.py` (and, pending
confirmation, `evidence.py`).

## 4. `run_pipeline` target shape (post-cutover)

```
translators/  +  normalize.py  +  owner_name_patterns.py
        │  (raw events, canonical_doc_type populated)
        ▼
   §17 debtor_party_engine        → debtor_resolved records
        ▼
   matcher.py  (§13.14 parcel resolution — see R2 for placement)
        ▼
   §18 leads_base_writer          → <source>_leads_base.json
        ▼
   §19 aggregator                 → matched_leads.json   ◀── immutable output of record
        ▼
   §20 semantic_verify            → deploy verdict
        │
        ├─ DEPLOY_BLOCKED          → HALT; pipeline stops, nothing scored or deployed
        ├─ NEEDS_OPERATOR_REVIEW   → proceed only on explicit operator approval
        └─ DEPLOY_OK               → proceed
        ▼
   SEAM  (matched_lead → scoring input: doc-type bridge G1, pattern G2,
          stack adaptation G3, enrichment join G4/G5)
        ▼
   score.py + classify.py + title-complexity   → scored_lead records (scored_leads.json)
        ▼
   review.py                      → review_flags / lead_status on scored_lead
        ▼
   dashboard.py                   → dashboard data.json
        ▼
   manifest.py                    → run manifest
```

The **§20 verdict is the gate**: scoring and everything below it runs only when
§20 returns `DEPLOY_OK` (or `NEEDS_OPERATOR_REVIEW` with approval). The §19
output `matched_leads.json` is never mutated by anything below it.

## 5. Golden-path test plan

The current `test_golden_path.py` (521 lines, 46 checks) is a self-contained
9-layer test (raw → normalize → match → scored lead → evidence → review →
dashboard → manifest → heartbeat) that **re-implements** each layer inline.
Post-cutover it is rewritten to drive the **real** modules, layer by layer,
without losing coverage:

| Old layer | New target | Coverage preserved |
|---|---|---|
| 1 raw record | `raw_event_record` | id / change-status / parser-confidence assertions |
| 2 normalize | real `normalize.py` + §17 `debtor_party_engine` | canonical doc type; debtor (not filer) resolved |
| 3 parcel match | `matcher.py` + §18 leads-base | `parcel_resolution_status` |
| 4 stacked lead | §18 + §19 aggregation + seam + real `score.py` / `classify.py` / title | **score, tier, score_reasons, deal_paths, title_complexity — asserted on real `scored_lead` output** |
| 5 evidence | `evidence_ledger.py` + traceability | evidence id / grade / linkage |
| 6 review | real `review.py` on `scored_lead` | review_flags, lead_status transition |
| 7 dashboard | real `dashboard.py` on `scored_lead` | display fields, Two-Truths invariant |
| 8 manifest | `manifest.py` | counts reconcile |
| 9 heartbeat | retained heartbeat | source health |

The scoring/classification/dashboard layers keep full coverage **because those
stages are retained** — the rewritten test asserts on their real output rather
than on inline re-implementations. The `matched_lead.signals` field follows the
§18 rich aggregated-group shape (F-8), not the §09 `signal_id`-string shape;
**§09 `09_output_schemas.md` needs a reconciliation note** stating the lead
`signals` field is the §18 shape. The Session 5 `test_staged_pipeline_end_to_end.py`
already covers §17→§20; the rewritten golden path extends it through the seam,
scoring, and the dashboard.

## 6. Cutover sequence (Part 2 — ordered)

1. **Build the seam adapter** — `canonical_doc_type → normalized_doc_type`
   bridge (G1), `pattern` derivation (G2), matched_lead `signals[]` → `stack`
   adaptation (G3); resolve the enrichment/attributes source (G4/G5, see R3).
2. **Define `scored_lead_record`** — new schema + dataclass mirror (Option Y).
3. **Implement the scoring stage** against `scored_lead` — wrap the retained
   `score.py` / `classify.py` / title-complexity, fed by the seam.
4. **Rewire `run_pipeline`** to the §4 shape; wire the §20 verdict gate.
5. **Adapt `stack.py`** input from raw signals to §19 matched-lead signals.
6. **Prove the full pipeline green end-to-end on synthetic data** — raw →
   §17→§18→§19→§20 → seam → scoring → dashboard — **before retiring anything.**
   This is the gate; a red end-to-end run stops the cutover.
7. **Rewrite `test_golden_path.py`** to the new architecture (§5); `run_all.py`
   green; county-agnostic scanner green.
8. **Only now retire the monolith core** — remove `build_leads.py`'s
   signal→identity→aggregation orchestration (and `evidence.py` if confirmed
   unused); archive cleanly (no dead second pipeline).
9. **Bump `FRAMEWORK_VERSION.json` → v5.4.0**; write `VERSION_NOTES_v5.4.0.md`.
10. **Commit; merge `feat/v5.4.0-pipeline-engine` → `main` (`--no-ff`); tag
    `v5.4.0` on the merge commit; push `main` and the tag.**

The load-bearing ordering: **step 6 (prove green end-to-end) and step 7 (golden
path green) precede step 8 (retire the monolith core).** The monolith stays the
safety net until the staged-plus-retained pipeline is proven.

## 7. Open risks / questions for the operator

- **R1 — input-side namespace.** `normalize.py` produces UPPERCASE
  `normalized_doc_type`; the §17 engine consumes lowercase per-county
  `canonical_doc_type`. The staged engine's *input* seam (translators /
  `normalize.py` → `raw_event_record.canonical_doc_type`) likely needs the same
  doc-type bridge as G1. Confirm whether `normalize.py` is updated to emit
  `canonical_doc_type`, or an input adapter is added. Part 1 scoped the
  downstream seam; this is flagged so Part 2 does not discover it late.
- **R2 — parcel resolution placement.** `matcher.py` (address→parcel) must run
  somewhere — most naturally between §17 and §18, so `leads_base` sees a
  populated `parcel_id`. Confirm placement.
- **R3 — enrichment / attributes (largest gap).** Scoring's attribute bonus and
  the dashboard's parcel display need parcel-master enrichment data (G4/G5).
  The staged pipeline does no enrichment. Options: (i) the seam invokes a
  retained enrichment stage; (ii) enrichment is a stage between §19 and the
  seam; (iii) scoring runs with attributes absent (degraded scores) until
  enrichment is wired. This needs an operator decision before Part 2.
- **R4 — `evidence.py` retirement.** Confirm no retained stage imports it before
  removal.

## 8. Recommendation summary

Adopt **Option Y** — scoring emits a new `scored_lead` record; `matched_leads.json`
stays immutable. Proceed with the Part 2 cutover sequence in §6, with the
operator first resolving R3 (enrichment/attributes sourcing). Do not retire the
monolith core until the full pipeline is proven green end-to-end (step 6) and
the golden path is rewritten and green (step 7).

**HALT — awaiting operator approval of the seam option (X / Y / Z) and a
decision on R3 before Part 2 executes.**
