# 06 — Hallucination Controls

The framework's worst failure mode is fabricating data that looks real. A leak rate of 1 hallucinated lead per 1,000 destroys the operator's reputation with their client, because that one fake lead will be the one the client tries to call.

This file defines what counts as fabrication, how the framework detects it, and how it prevents export.

---

## What counts as a hallucinated claim

Any of the following:

1. **A field is filled with a plausible-looking value not derived from a primary source.** Example: an "owner name" generated by inference when the parcel master had a redacted owner field.
2. **A pattern fires from inferred metadata rather than a recorded event.** Example: a `transfer` pattern firing from a $1 deed in the parcel master with no clerk DEED record to confirm.
3. **A score is assigned without traceable reasons.** Every score must have a reasons array with arithmetic that adds up to the final score.
4. **A deal-path classification fires without rationale.** Every deal path must carry a rationale string referencing the patterns and attributes that triggered it.
5. **A status label is upgraded.** A field labeled `Estimated` does not become `Confirmed` between pipeline runs unless a primary source is added. Status downgrades (Confirmed → Estimated when source becomes unavailable) are allowed; upgrades require new evidence.
6. **A source URL is generated.** URLs are copied verbatim from scraper output. The framework never constructs URLs from templates without verification.
7. **A claim about an owner's intent is made.** "Owner is motivated to sell" is not a verifiable claim. The framework reports signals; it does not interpret intent.
8. **A canonical document type is invented.** The AI does not create new entries in `canonical_doc_types.json` during a build, and does not produce `normalized_doc_type` values that don't exist in the registry. Unknown raw labels route to review with `unknown_doc_type` per `domain/05_review_queue_rules.md`. The registry grows only through deliberate operator action, not autonomous AI inference.
9. **A lifecycle stage is fabricated.** The AI does not infer that a property is at a particular stage of a foreclosure / probate / lien lifecycle without recorded events supporting the stage. Stage advances when a recorded canonical type fires, never on inference.
10. **A title complexity score is asserted without contributors.** Every title complexity score must carry a `contributors` array showing which factors and weights produced it. A bare numeric score with no contributors is fabrication.

## Hallucination warning signs in AI output

When operating autonomously, the AI must self-check for these patterns. If any fires, stop and consult `07_fallback_metrics.md`.

- Confident claim without source URL
- Filled field where the source was empty or redacted
- "I think this is..." or "this looks like..." in a build summary (these are inference cues, not facts)
- Investment outcome claimed (ROI estimates, profit projections, "good deal" labels) without analysis showing derivation
- Scraper claimed working without a test run output
- Property match assigned without confidence score
- Export executed without dedupe pass
- Test/synthetic data mixed into real data in `leads.json`
- Review queue skipped on data flagged for review
- Scoring weights changed without operator approval

When the AI catches itself doing any of these, it stops the current action and routes to fallback.

## Hallucination risk score

Every record gets a hallucination risk score, computed at build time:

```
risk_score = 0
+ 20 if source_url is missing
+ 20 if any required field is missing (owner, parcel_id, mailing_address)
+ 20 if match_confidence < 80
+ 20 if doc_type is unmapped
+ 30 if any field carries Unsupported label
+ 15 if any field carries Estimated label without derivation array
+ 50 if investment outcome is claimed without analysis
+ 50 if synthetic data is co-mingled with real data
```

Risk levels:

| Score | Level | Action |
|---|---|---|
| 0–20 | Low | Allow export |
| 21–49 | Medium | Add to review queue |
| 50–79 | High | Block export, route to review with high-priority flag |
| 80+ | Critical | Halt build, require operator approval before continuing |

A `Critical` risk score causes the build to fail. The framework writes `BUILD_BROKEN.md` and exits non-zero. This is the hard floor.

## The Knowledge Base Guardian role

When the AI runs autonomously, it operates a Knowledge Base Guardian role concurrent with the build. The Guardian's job is to monitor the build's outputs against this knowledge base and stop unsupported claims before they reach `leads.json`.

The Guardian checks:

1. Every record in `leads.json` has a source URL — fail otherwise
2. Every record has a deal-path classification — fail otherwise
3. Every score has a reasons array — fail otherwise
4. Every claim labeled `Confirmed` is backed by a scraper log line that captured it from source
5. Every claim labeled `Estimated` has a `derivation` field showing the formula or proxy used
6. No record has fields that are mathematically inconsistent (e.g., assessed_value > land_val + imprvt_val by more than rounding)

When the Guardian finds a violation, it:
- Re-labels the offending field to `Unsupported` or `Unknown`
- Adds the record to review queue
- Logs the violation to `data/raw/guardian.log`
- If 5+ violations on a single build, emits Telegram alert

## Status labels — the language of uncertainty

Every field in a lead record must carry a status label. Allowed labels:

| Label | Meaning |
|---|---|
| `Confirmed` | Verified by a primary source (scraper log captured it) |
| `Estimated` | Derived from a model or proxy; derivation is recorded |
| `Possible` | Pattern matches but not verified |
| `Needs Review` | Flagged for human eyes |
| `Unknown` | Field expected but missing |
| `Unsupported` | Claim was made but no source backs it |
| `Blocked` | Source unreachable; no current value |
| `Do Not Export` | Failed one or more fallback metrics |

These labels travel with the record through the pipeline. The dashboard renders them as small icons next to fields that aren't `Confirmed`, so the operator's client sees uncertainty at a glance.

The CSV export includes a column per field showing its status label, so when the lead lands in the client's CRM, the client knows which fields to verify before calling.

## Examples of hallucination prevention

### Example 1: Missing owner name

**Source:** parcel master shows `OWNER_NAME = ""` (state-level redaction policy on the source).

**Wrong:** `owner_name = "Likely individual based on property class"`

**Right:** `owner_name = null, owner_name_status = "Unknown"`, lead routes to review with reason `party_missing` if no other source provides the name.

### Example 2: Inferred sub-to candidacy

**Source:** parcel master shows last sale 2021-03 for $450K. No mortgage data.

**Wrong:** `sub_to candidate, confidence: high — favorable rate era`

**Right:** `sub_to candidate, confidence: moderate — last sale 2021-03 (favorable rate era proxy); mortgage data unavailable, payment status unverified`. Rationale explicitly notes the proxy and the missing data.

### Example 3: Generated source URL

**Source:** sheriff PDF row mentions "Case CH 780549."

**Wrong:** `source_url = "https://courts.<target-state-domain>/cases/CH-780549"` (constructed from template)

**Right:** `source_url = "https://<county-clerk-domain>/WebContentFiles/<actual sheriff PDF URL>"` (the PDF where the case was actually found), `source_url_status = "Confirmed"`. The case-detail URL is left blank because no scraper verified it.

### Example 4: Investment outcome claim

**Wrong:** `notes: "This is a good wholesale deal — high equity, motivated seller"`

**Right:** `notes` does not contain investment-outcome language. The deal-path classifier emits `wholesale candidate, confidence: high` with rationale. The operator's client makes the investment judgment.

## What the AI must NEVER do

When operating autonomously and faced with sparse data, the AI must NOT:

- Fill missing fields by guessing plausible values
- Claim a record is verified when only one source provides it
- Generate URLs from templates
- Estimate investment outcomes (ARV, repair costs, profit) without an analysis showing inputs
- Mark records as `Confirmed` to make the build look more complete
- Skip the review queue to reduce manual workload
- Mix synthetic test data with real data in production `leads.json`
- Continue a build when hallucination risk goes Critical

The framework's value comes from honest pipelines. A pipeline that hallucinates 1% of leads is worse than no pipeline, because the operator's client will trust it and call a fake.

---

## Operator override

If the operator explicitly asks the AI to do something this file forbids (e.g., "fill in the owner names by inference"), the AI must refuse and explain why. This is one of the pushback cases listed in the master prompt.

The AI does not silently comply with hallucination requests. It states the rule, declines, and offers the supported alternative (e.g., skip-trace vendor integration, public-records owner-name request).
